# ADRs

Architecture Decision Records in this directory capture meaningful long-term
constraints introduced by the repository or product design.

Create or update an ADR when a change introduces a repository-wide rule,
durable tradeoff, or long-lived constraint that future agents should not need
to rediscover from chat history or commit archaeology.

Good ADR candidates include:

- persistence or migration strategy
- provider contract shape
- Anki identity or idempotency rules
- prompt versioning policy
- storage model changes

Keep each ADR concise and include:

1. Status
2. Context
3. Decision
4. Consequences
5. Alternatives considered

Do not create ADRs for small refactors, local naming choices, or temporary task
state.
